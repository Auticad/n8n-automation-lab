1 Panoramica

Il workflow espone un endpoint HTTP webhook che accetta qualsiasi payload JSON, lo classifica tramite Google Gemini 2.5 Flash in una delle cinque categorie (informational, transactional, error, alert, other), e scrive il risultato su Google Sheets. Opera in response mode sincrono: il client aspetta la risposta HTTP fino al completamento dell'esecuzione.

Introduce tre pattern avanzati assenti nei workflow precedenti: il retry loop con contatore per gestire il rate limiting di Gemini (429), il dead-letter pattern per tracciare gli errori non recuperabili su un tab Sheets separato, e il multi-branch error routing con un nodo Respond to Webhook dedicato per ogni codice HTTP di uscita.

2 Architettura a layer

Il workflow è organizzato in 5 layer funzionali. Ogni layer ha responsabilità distinte e confini netti — il dato si muove in avanti, gli errori si gestiscono localmente con dead-letter e risposta dedicata.

7 Osservazioni tecniche
Markdown stripping — comportamento Gemini 2.5 Flash

Il modello risponde con il JSON avvolto in backtick markdown (` ```json ... ``` `) anche quando esplicitamente istruito a non farlo. Il Code_ParseGeminiResponse rimuove le decorazioni prima del parsing. Se il problema persiste, valuta l'uso di response_mime_type: "application/json" nel body della richiesta — disponibile su Gemini 1.5+ e 2.x, forza una risposta JSON pura senza decorazioni.

Set_Enrich — includeOtherFields è critico

Il nodo Set in modalità assignments elimina tutti i campi non esplicitamente dichiarati. Senza includeOtherFields: true, i campi headers e body del webhook vengono persi, e il nodo di validazione non trova il Content-Type — risponde 415 a qualsiasi richiesta. Verifica questo toggle dopo ogni importazione.

Loop retry — back-edge non esprimibile nel JSON SDK

Il collegamento Wait_60s → HTTP_Classifier_Gemini crea un ciclo nel grafo del workflow. n8n supporta i cicli ma il formato SDK non lo esprime — va aggiunto manualmente nel canvas. Il workflow è funzionale anche senza: dopo il terzo 429 va in dead-letter normalmente.

Pattern dead-letter — tracciabilità completa degli errori

Ogni errore non recuperabile produce una riga sul tab Errors con timestamp, tipo, messaggio e payload originale (max 5.000 char). Questo rende il sistema osservabile: è possibile costruire un report degli errori direttamente su Sheets senza accedere ai log n8n.

Response mode sincrono — timeout 30 secondi

Con responseMode: responseNode n8n tiene aperta la connessione HTTP per tutta la durata dell'esecuzione. Il timeout di Gemini è configurato a 30s, ma con 3 retry da 60s ciascuno il tempo massimo teorico supera i 3 minuti. In produzione su volumi significativi valuta di passare a un pattern asincrono: webhook + risposta immediata 202 + elaborazione in background + notifica.
