1. Panoramica

Il workflow riceve tramite form pubblico una descrizione testuale del video, uno stile visivo e un'immagine del prodotto da usare come riferimento. Dal trigger si diramano immediatamente due branch paralleli: uno genera il prompt video in inglese tramite gpt-4o, l'altro converte l'immagine in base64 e la carica su un server temporaneo KIE AI per ottenerne un URL pubblico.

I due branch vengono ricombinati e il workflow chiama l'API KIE AI Veo 3.1 per avviare la generazione video in modalità REFERENCE_2_VIDEO — il video viene generato usando l'immagine come riferimento visivo. La generazione video non è sincrona: il workflow implementa un polling loop che interroga l'API ogni 60 secondi fino a quando il campo data.completeTime non è valorizzato, indicando il completamento.

Tecnologie principali: n8n Form Trigger (con upload file binario) → gpt-4o LLM Chain (generazione prompt) + Code JS (base64 extraction) → KIE AI file upload API → KIE AI Veo 3.1 video generation API → polling loop (Wait + If).

2
Code in JavaScript
Code (Branch B)
ParametroValore
LinguaggioJavaScript
Outputbase64_string, mime_type, original_field_name
// Legge i byte reali del file dal filesystem interno di n8n
const items = [];

for (let i = 0; i < $input.all().length; i++) {
  const item = $input.all()[i];
  const binaryKeys = Object.keys(item.binary || {});

  if (binaryKeys.length === 0) {
    items.push({ json: { ...item.json, error_debug: "Nessun dato binario trovato." } });
    continue;
  }

  const firstKey = binaryKeys[0];
  const buffer = await this.helpers.getBinaryDataBuffer(i, firstKey);
  const base64 = buffer.toString('base64');
  const mimeType = item.binary[firstKey].mimeType;

  items.push({
    json: {
      ...item.json,
      base64_string: base64,
      mime_type: mimeType,
      original_field_name: firstKey
    }
  });
}

return items;
Importante: con binaryMode: "separate" (n8n salva i binari su filesystem) il campo item.binary[key].data non contiene il base64 ma solo un riferimento interno. Per i byte reali serve await this.helpers.getBinaryDataBuffer(i, key) seguito da .toString('base64'). Il codice trova da solo il primo campo binario, quindi resta valido anche se il nome del campo nel form cambia.

REFERENCE_2_VIDEO: la modalità usa l'immagine del prodotto caricata come riferimento visivo per guidare la generazione del video — il soggetto nel video sarà coerente con l'immagine fornita. La risposta contiene data.taskId usato nel polling.
veo3_fast: versione più veloce (e meno costosa) del modello. Usare veo3 per qualità massima, con tempi di attesa più lunghi (il Wait potrebbe dover essere aumentato).

Attenzione ai loop infiniti: se l'API KIE AI restituisce un errore o il task fallisce, completeTime potrebbe non essere mai valorizzato e il loop non termina. Considera di aggiungere un contatore di iterazioni massime (es. 20 tentativi = 20 minuti) con un nodo Stop and Error.

ParametroValore

Response FormatFile (opzione in Response → Response Format)

Perché serve questo nodo: il nodo Gmail accetta allegati solo come dati binari presenti nell'item — non accetta URL stringa. Impostando Response Format = File, l'HTTP Request scarica il contenuto del video e lo salva nel campo binario data, che Gmail può allegare direttamente.
Errore frequente: passare {{ $json.data.response.resultUrls[0] }} direttamente al campo "Attachment Field Name" di Gmail produce l'errore "binary file not found" perché n8n cerca un campo binario con quel nome nell'item (che non esiste). Il campo deve essere sempre il nome del binario — in questo caso data.

4. Prerequisiti
n8n Istanza locale o cloud. Versione ≥ 1.x per i nodi LangChain. Il Form Trigger richiede che n8n sia accessibile pubblicamente (o via tunnel) per mostrare il form.
OpenAI API Key Credenziale OpenAI API nativa per il nodo LLM Chain (gpt-4o).
KIE AI API Key Chiave API per kie.ai. Usata in entrambi gli HTTP Request KIE AI (upload immagine + submit video). Attenzione: l'URL di upload usa un dominio differente (kieai.redpandaai.co).
Account KIE AI con crediti La generazione video Veo 3.1 consuma crediti API. Verifica il saldo su kie.ai/dashboard.
5. Configurazione prima dell'uso
  1. Importa video-veo31-kie-ai.json in n8n (Workflow → Import from file)
  2. Crea una credenziale OpenAI API con la tua chiave sk-... e collegala al nodo OpenAI Chat Model
  3. Apri il nodo HTTP Request (upload immagine) e imposta il Bearer token nell'header Authorization con la tua chiave KIE AI: Bearer KIE_YOUR_API_KEY
  4. Apri il nodo Submit_Veo_task e imposta lo stesso Bearer token nell'header Authorization
  5. Apri il nodo HTTP Request1 (status check) e imposta lo stesso Bearer token
  6. Rimuovi il pin data dal nodo Basic LLM Chain e dal nodo Merge (click destro sul nodo → "Unpin") per consentire l'esecuzione reale
  7. Attiva il workflow → n8n genera l'URL pubblico del form — condividilo o aprilo in un browser
  8. Compila il form con descrizione, stile e immagine prodotto → osserva l'esecuzione in n8n
Credenziali nei nodi HTTP: questo workflow usa il pattern manuale (header hardcoded nel nodo) invece delle credenziali n8n native. Il Bearer token è visibile nel JSON. Nel file sanitizzato del repository il valore è già rimosso — inserisci la tua chiave solo in n8n, non nel file JSON committato.
6. Concetti chiave appresi

Polling loop in n8n: quando un'API esterna è asincrona (restituisce un task ID invece del risultato immediato), il pattern standard è: submit → wait → check → if done: fine, else: wait again. In n8n si implementa con Wait + HTTP Request + If con il ramo False che torna al nodo HTTP.

Estrazione dati binari da form: quando un Form Trigger riceve un file, il contenuto arriva nel campo binary dell'item, non nel json. Per accedervi serve un nodo Code che usa item.binary[fieldName].data. L'approccio di questo workflow (iterare i binaryKeys senza hardcodare il nome) è più robusto di accedere direttamente al nome del campo.

Pin data per sviluppo: i nodi con pin data saltano l'esecuzione reale e usano i dati salvati. È utile in fase di test per non consumare crediti API ad ogni esecuzione. In produzione va sempre rimosso. Il nodo LLM Chain e il nodo Merge in questo workflow hanno pin data attivi — segno che il workflow è stato debuggato prima del deploy finale.

REFERENCE_2_VIDEO: la modalità di generazione che usa un'immagine di riferimento per guidare il video è la funzionalità differenziante di questo workflow rispetto a una generazione video text-only. Richiede che l'immagine sia disponibile come URL pubblico — da qui il passaggio di upload intermedio su un server KIE AI temporaneo.

Estensione suggerita: aggiungere dopo il nodo If (output True) un nodo che invia l'URL del video via email o Telegram. La risposta dell'API KIE AI al completamento dovrebbe contenere data.videoUrl o simile — controlla la documentazione kie.ai/docs per il campo esatto da estrarre.
